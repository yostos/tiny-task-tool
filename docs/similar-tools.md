# 類似ツール調査

Updated: 2026-01-20

ターミナル環境で動作するタスク管理ツールの調査結果です。

---

## [Taskwarrior](https://taskwarrior.org/)

- **UI形式**: CLI（[taskwarrior-tui](https://kdheepak.com/taskwarrior-tui/)で TUI 拡張可能）
- **データ形式**: 内部データ（JSON エクスポート対応）
- **特徴**:
  - 2006年から開発されている老舗ツール
  - 非常に強力なフィルタリングとレポート機能
  - プロジェクト階層（ドット記法: `Home.Kitchen`）
  - タスク間の依存関係
  - Taskserver による複数デバイス同期
  - 豊富なプラグインエコシステム
- **TTT にはない機能**:
  - 高度なフィルタ・クエリ言語
  - 再発タスク（recurring tasks）
  - タスク同期（Taskserver）
  - カスタムレポート定義
  - UDA（User Defined Attributes）

---

## [simple-task-master](https://github.com/carlrannaberg/simple-task-master)

- **UI形式**: CLI
- **データ形式**: Markdown（個別ファイル、YAML フロントマター付き）
- **特徴**:
  - 各タスクが独立した Markdown ファイル
  - YAML フロントマターでメタデータ管理
  - タグ、優先度、期限日
  - 複数出力形式（JSON, CSV, テーブル）
  - Node.js 製
- **TTT にはない機能**:
  - タスクごとの独立ファイル管理
  - タグによるフィルタリング
  - 優先度設定
  - 期限日管理
  - JSON/CSV エクスポート

---

## [TTDL (Terminal ToDo List)](https://github.com/VladimirMarkelov/ttdl)

- **UI形式**: CLI
- **データ形式**: todo.txt（プレーンテキスト）
- **特徴**:
  - Rust 製で高速
  - 標準的な todo.txt 形式に準拠
  - プロジェクト（`+`）とコンテキスト（`@`）による分類
  - 人間が読める日付形式（`due:mon`, `due:1w`）
  - シンタックスハイライト
- **TTT にはない機能**:
  - 時間トラッキング（`start`/`stop` コマンド）
  - 再発タスク（`rec:1w` で毎週など）
  - 閾値日（threshold date）
  - 統計レポート（`stats` コマンド）
  - プラグインシステム

## [Taskdog](https://www.geekfeed.co.jp/geekblog/wada-taskmanagement-tool)

- **UI形式**: CLI + TUI + REST API
- **データ形式**: JSON（内部）
- **特徴**:
  - Clean Architecture で設計
  - 依存関係とタグによるタスク構造化
  - vi キーバインド対応の TUI
  - リアルタイムフィルタリング
  - MCP Server 連携（Claude Desktop 統合）
- **TTT にはない機能**:
  - 9種のスケジューリングアルゴリズム
  - 自動タスク配置（休日・週末回避）
  - ガントチャート表示
  - REST API
  - WebSocket によるリアルタイム通知
  - AI 連携（MCP Server）

## TTT の差別化ポイント

上記ツールと比較した TTT の特徴：

1. **ノード指向**: 階層タスクをネイティブサポート。親タスク完了で子も完了、アーカイブも一括移動。他ツールは依存関係やプロジェクト階層で代替するか、サードパーティ拡張が必要。
2. **1ファイル制約**: ファイル選択の判断を排除。「今日」に集中する設計。
3. **外部エディタ連携**: 編集は慣れたエディタに委ねる Unix 哲学。
4. **機能の少なさ**: 検索、タグ、優先度、同期などを意図的に除外。

## 機能取り込みの検討結果

### 結論

**現状のロードマップを維持し、新たな機能追加は見送る。**

concept.md の設計思想「機能を足すことより、削ることに価値がある」に従い、類似ツールからの追加機能取り込みは行わない。

### 採用しない機能

| 機能 | 理由 |
|------|------|
| 再発タスク | 実装が複雑、シンプルさを損なう |
| タグ・優先度 | 「どのタグにしよう」という判断が発生する |
| フィルタリング | 「一枚の紙」を複雑にする |
| 閾値日 | 「今日」ではなく「未来」の管理になる |
| 時間トラッキング | Unix哲学に従い外部ツール（Timewarrior等）に任せる |
| プラグイン | メンテナンス負荷増大 |
| API/AI連携 | スコープ外 |

### @due（期限）について

ロードマップ v0.5.0 で検討中の `@due` は採用する方針。

TTT は「その日にやるべきことを書くツール」ではなく、**「やるべきと思いついた時にすぐに書き留めるツール」** である。したがって：

- 数日前に書き留めた TODO が残っている可能性がある
- 「今日はどの TODO を片づけようか」と選ぶ際に、@due は重要な判断材料となる
- 「今日」に集中するコンセプトと矛盾しない。むしろ「今日、何をすべきか」を判断するための情報として機能する
